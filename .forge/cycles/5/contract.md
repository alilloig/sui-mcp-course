# Cycle 5 Contract — `help-ladder-three-rungs`

## Behavior

After this cycle, the plugin's MCP server exposes the seventh tool spelled out in spec.md `## MCP Tools` — `requestHint({ projectRoot, rung })` — and the engine implements the full three-rung help ladder per spec.md `## Help-Ladder Protocol`. The skill drives it inside the `course-conductor` agent's spot loop (cycle 4 wired the loop; cycle 5 fills in the failure-routed branch): when a `verifySpot` call returns `pass: false`, the agent surfaces the rung-1 hint offer; the student opts in, the agent calls `requestHint({ rung: 1 })`, which returns the rendered `hint.md` payload (with `{{ ... }}` substituted from `state.personalization` via `personalization.ts`'s `substitutePromptOnly`) and flips `state.ladder[spot.id].hint_used` to `true`. After another `verifySpot` failure (or unconditional opt-in), the agent calls `requestHint({ rung: 2 })`, which returns `reference.md` and flips `reference_shown`. After yet another failure, `requestHint({ rung: 3 })` performs the load-bearing rung: it (1) reads the rung-3 markdown payload from `paths/<slug>/rungs/<spot.id>/auto.md`, (2) reads `<projectRoot>/<spot.target_file>` and slices out the `target_range` byte slice, (3) snapshots that slice to `<projectRoot>/.sui-deepbook-course/snapshots/<spot.id>.bak` (creating the directory; rotating any existing `.bak` to `.bak.<ISO-timestamp>` first so prior rung-3 attempts stay recoverable), (4) overwrites the range with the substituted payload by rewriting the whole `target_file` once, (5) flips `auto_completed` (set on rung-3 *invocation*, never cleared — AC-5.4) and saves state, then (6) dispatches `runVerifySpot` against the just-written file. On verification pass, the cursor advances exactly as in cycle 4's pass branch (one `advanceCursor` + `saveState`), and the `requestHint` response carries `{ ok: true, payload, newLadder, autoVerifyResult: { pass: true, advanced: true } }`. On verification fail, the cursor is left untouched, `auto_completed` stays set permanently (the student can keep editing and call `verifySpot` again), and the response carries `{ ok: true, payload, newLadder, autoVerifyResult: { pass: false, advanced: false, output? } }`. Re-attempts after a failed rung-3 are explicitly supported: the next `verifySpot` call sees the same cursor, runs the standard `runVerification` adapter, and on pass advances exactly as it would for any other spot. AC-5.1 (rung gating), AC-5.2 (rung mutates state), AC-5.3 (auto-complete advances only on pass), AC-5.4 (`auto_completed` permanent) are all unit-testable invariants the cycle owns.

Rung gating is enforced at the `requestHint` tool boundary as a structured error, not silently. Rung 2 requires `state.ladder[spot.id]?.hint_used === true`; rung 3 requires `state.ladder[spot.id]?.reference_shown === true`. Out-of-order calls return `{ ok: false, error: { kind: 'rung-out-of-order', requestedRung, requiredPriorRung, missingFlag } }` with no state mutation, no filesystem mutation, and no auto-write. A new `mcp/server/src/ladder.ts` engine seam owns the rung-state mutations and the rung-3 side effect: it exposes `recordRungUse(state, spotId, rung) → State` (returns a new state with the requested rung's flag set to `true` plus existing flags preserved — append-only; reviewers grep for any caller that sets a ladder flag back to `false`), `canAdvanceRung(state, spotId, rung) → { ok: true } | { ok: false; missingFlag: string; requiredPriorRung: 1 | 2 }`, and `runAutoWrite(projectRoot, spot, payload) → Promise<{ backupPath: string; bytesWritten: number }>` which performs the snapshot-then-overwrite step. The auto-write target is the line range described by `spot.target_range` (`'<startLine>-<endLine>'`, 1-indexed inclusive — e.g. `'39-58'` rewrites lines 39 through 58); the snapshot file path is the literal `<projectRoot>/.sui-deepbook-course/snapshots/<spot.id>.bak` (substitution-free; `${spot.id}` is the manifest slug, not a personalization key) and the directory is `mkdir -p`'d on first use. Snapshot files are append-only across rung-3 attempts (the second rung-3 invocation against the same spot rotates the existing `.bak` to `.bak.<ts>` before writing the new snapshot), so a student who hits rung 3 twice still has every prior version recoverable. None of `requestHint`'s code paths emits `action.kind === 'shell'` (cycle 1 A14 / cycle 3 A12 / cycle 4 A11(a) carry-forward); the auto-write happens in-process via `fsPromises.writeFile`, not via the runPreflightProbe shell channel. Rung-3 step ordering inside `requestHint` is: (a) read rung markdown FIRST (so `rung-content-missing` short-circuits before any side effect); (b) snapshot the existing range; (c) overwrite the target file; (d) flip `auto_completed` to `true` and save state; (e) dispatch `runVerifySpot`. AC-5.4 holds because `auto_completed` is set as soon as the rung-3 read+write+flip atomic block completes — the only way to "not set" it is for the read or snapshot or overwrite to fail before the flag flip, in which case the rung-3 invocation never produced a side effect to commemorate.

This cycle bundles four cycle-4 carry-forwards and one cycle-3 carry-forward because their files are already in scope or trivially adjacent and they tighten invariants the new tool depends on. **(M001 from cycle-4 review)** wrap `runVerifySpot`'s call to `runVerification` in a try/catch that catches `VerificationModeUnsupportedError` and surfaces `{ pass: false, error: 'verification mode <mode> not yet supported', advanced: false }` — cycle 5's `requestHint` rung-3 path threads the same dispatcher and would otherwise propagate the throw out the MCP boundary. **(M002 from cycle-4 review)** wrap every `saveState` call in `selectPath`/`setPersonalization`/`verifySpot`/`requestHint` in try/catch that surfaces a `state-save-failed` warning kind (carries the original error message) and returns the appropriate tool-shape failure (`{ ok: false, errors: [...] }` for selectPath/setPersonalization; `{ pass: true, advanced: false, error: 'verification passed but state persist failed: ...' }` for verifySpot's pass-but-save-fails branch; `{ ok: false, error: { kind: 'state-save-failed', message: ... } }` for requestHint). **(L002 from cycle-4 review / cycle 1 AC-1.3 carry-forward)** add the outputStyleOk gate at the new `requestHint` tool boundary AND retroactively at `selectPath`/`setPersonalization`/`nextSpot`/`verifySpot` — each tool short-circuits with a structured `output-style-disabled` failure when the cycle-1 settings.json check returns `outputStyleOk: false`, so AC-1.3's zero-write invariant holds across the *full* tool surface, not just `start`. **(L001 from cycle-4 review)** replace `command.split(' ')` in verify.ts's compile adapter with a small shell-style parser (`parseCommand(cmd) → { cmd, args }`) that respects double-quoted segments and collapses runs of whitespace; curriculum authors will hit the naive split when a phase's `build_command` becomes anything beyond `pnpm build`. **(R3-005 from cycle-3 review)** add `AbortSignal.timeout(5000)` to `checkManifest`'s fetch in `mcp/server/src/probes/manifest.ts` so a hung port-9009 binding can't stall the probe indefinitely.

Two carry-forwards stay explicitly **deferred**. The cycle-2 C008 `state-shape-invalid` kind distinct from `state-corrupt` stays deferred — cycle 5 makes no breaking schema changes (`auto_write_attempted` already exists in the LadderRung schema from cycle 4's append-only addition), so STATE_SCHEMA_VERSION stays at 1 and the corruption-vs-shape distinction is not yet load-bearing. The cycle-4 M002 cursor-done sentinel (`'__done__'`) likewise stays deferred — cycle 5's rung-3 only fires when the cursor is on a real spot (gating excludes the done state via `getCurrentSpot.done === false`), and bumping the schema for that hardening is out of scope here. Cycle 1+2+3+4 invariants stay intact: `"01-orderbook-viewer"` outside `paths/` and `tests/fixtures/` remains zero (A11(b)/A21 from cycle 4); `setSpawnOverride`/`setVerifyOverride` symbols remain absent (cycle 4 A13 and the H001 remediation); `kind: 'shell'` outside `runPreflightProbe.ts` stays at zero strings; `personalization.ts`'s `substitutePromptOnly` remains the only `{{ ... }}` resolver (cycle 4 A7); registry warning kinds remain exactly the eight `registry.ts` emits (cycle 4 A12 / M001); `state.ts` durability sequence stays at `open → handle.sync → handle.close → rename` (cycle 4 A14). The new `requestHint.ts` and `ladder.ts` files participate in those invariants from day one.

- `mcp/server/src/ladder.ts` — **New.** Engine seam for ladder mutations and the auto-write side effect. Exports: (a) `recordRungUse(state: State, spotId: string, rung: 1 | 2 | 3) → State` — returns a new `State` whose `ladder[spotId]` has the rung's flag set to `true` (`hint_used` for rung 1, `reference_shown` for rung 2, `auto_completed` AND `auto_write_attempted` for rung 3) plus any existing flags preserved, plus a default-all-false rung created if the spot has no ladder entry yet; the function is purely append-only (it never sets a flag from `true` to `false`); (b) `canAdvanceRung(state: State, spotId: string, rung: 1 | 2 | 3) → { ok: true } | { ok: false; missingFlag: 'hint_used' | 'reference_shown'; requiredPriorRung: 1 | 2 }` — rung 1 always returns `{ ok: true }`; rung 2 requires `ladder[spotId]?.hint_used === true`; rung 3 requires `ladder[spotId]?.reference_shown === true`; (c) `runAutoWrite(projectRoot: string, spot: SpotData, payload: string) → Promise<{ backupPath: string; bytesWritten: number }>` — parses `spot.target_range` (`'<startLine>-<endLine>'`, 1-indexed inclusive), reads `<projectRoot>/<spot.target_file>` (rejects `AutoWriteError { kind: 'target-file-missing' }` on ENOENT), splits on `\n`, slices out the range (rejects `AutoWriteError { kind: 'target-range-invalid' }` if range parse fails or is out of bounds), writes the existing range bytes to `<projectRoot>/.sui-deepbook-course/snapshots/<spot.id>.bak` (creates the snapshots/ directory; if the `.bak` already exists, rotate it to `.bak.<ISO-timestamp-no-colons>` *first*; rejects `AutoWriteError { kind: 'snapshot-write-failed' }` on rotate or write failure), then splices the substituted payload into the line array at the same range and rewrites the whole `target_file` via `fsPromises.writeFile` (rejects `AutoWriteError { kind: 'overwrite-failed' }` on write failure). The auto-write is *not* atomic across snapshot+overwrite — the snapshot completes first and is durable on disk before the overwrite begins, so a crash mid-overwrite leaves both the snapshot and the (partially written) target_file recoverable by hand. Snapshot files are written with `wx` flag and mode `0o600`, mirroring `state.ts` durability conventions. Reviewer grep target: `ladder.ts` contains exactly two `fsPromises.writeFile` call sites (one for `target_file`, one for the `.bak` snapshot path); no other write surface.
