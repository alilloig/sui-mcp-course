# Cycle 3 Contract — `preflight-eight-probes`

## Behavior

After this cycle, the plugin's MCP server exposes the eight ordered preflight probes spelled out in spec.md `## Preflight Checks (ordered)` through a single new tool, `runPreflightProbe`. The skill drives the probes one at a time in the spec table's order: (1) Docker running, (2) Node ≥ 18, (3) pnpm available, (4) Sui CLI in 1.63.2–1.64.1, (5) `sui-pilot@<source>` enabled, (6) `~/workspace/deepbook-sandbox/` repo present, (7) sandbox manifest reachable on `http://localhost:9009/manifest`, (8) `learning-output-style@claude-plugins-official` enabled. Each probe returns `{ pass, message, action? }`. Probe #7 is the only probe that may attach an `action.kind === 'shell'` payload (the `pnpm deploy-all --quick` remediation). Probe #8 reuses cycle 1's `outputStyle.ts` directly — no second copy of the settings-file parsing exists; the probe is a thin wrapper that calls `probeOutputStyle()` and re-classifies the result as a probe outcome. Probes #1, #4, #6 are guarded preconditions for the deploy remediation: even when the student grants explicit approval, the engine refuses to spawn `pnpm deploy-all --quick` if Docker is not running, the sandbox repo is absent, or the Sui CLI is out of range — preserving AC-2.4's hard stop and the spec's "no deploy without prerequisites" promise.

The deploy remediation has **two strictly-disjoint modes** behind a single env-var switch. Setting `E2E_DEPLOY_STUB=1` is the **sole** entry point to the stub branch (no other env var, no flag, no JSON config switch, no fallback). The stub branch is for the e2e harness's E-007 scenario (deploy-all that exits 0 but never brings the manifest up); it spawns a deterministic in-process stand-in instead of the real subprocess. When `E2E_DEPLOY_STUB` is unset (or any value other than `'1'`), the only path that triggers `pnpm deploy-all --quick` is the real-deploy branch: it requires explicit student approval through `runPreflightProbe({ probeId: 'sandbox-manifest-reachable', remediate: true })`, gates on probes #1/#4/#6 returning pass, spawns `pnpm deploy-all --quick` with `cwd: ~/workspace/deepbook-sandbox/sandbox/`, streams stdout/stderr lines back through the tool response, and enforces a hard timeout (default 7 minutes) that aligns with E-008's preconditions. After spawn, probe #7 re-runs up to three times; persistent failure is reported as a stop with the captured logs and a `pnpm down` suggestion. The cycle preserves cycle 1's AC-1.3 zero-write-on-output-style-fail invariant: `runPreflightProbe` does not write under `<projectRoot>/.sui-deepbook-course/` at all (preflight is read-only with respect to project state), and `start`'s output-style gate continues to short-circuit before any preflight is even offered.

This cycle also bundles three carry-forwards from cycle 2's review that the contract considers bounded enough to land here without expanding scope: (a) `saveState`'s O_RDONLY-fd fsync (C002+C007+C009+C027 — replace `fs.openSync(tmp, 'r')` + `fs.fsyncSync(fd)` with `fsPromises.open(tmp, 'r+')` + `handle.sync()` + `handle.close()`), (b) the missing `wx` flag and `mode: 0o600` on tmp/archive writes (C016), and (c) unifying `RegistryWarning` + `StateWarning` into a single discriminated `EngineWarning` union in `mcp/server/src/warnings.ts` (C017/C018) — this is the right cycle to unify because cycle 3 introduces a new warning surface (preflight) and folding three sources at once is cheaper than folding two now and three next cycle. None of the three bundles change wire shape: existing warning `kind` values are preserved verbatim; only their TypeScript home moves.

## Files

- `mcp/server/src/preflight.ts` — Ordered probe registry. Exports `PROBE_ORDER: readonly ProbeId[]` (literal-typed array of the eight probe ids, in spec table order) and `runProbe(probeId, { remediate?: boolean }) → Promise<ProbeResult>`. `ProbeResult` is `{ pass: boolean; message: string; action?: ShellAction }` where `ShellAction = { kind: 'shell'; command: string; cwd?: string; timeoutMs?: number }`. The registry is a `Record<ProbeId, Probe>` map; adding a probe is one entry plus one literal in `PROBE_ORDER`. Owns no probe logic itself — delegates to the per-probe modules below.
- `mcp/server/src/probes/docker.ts` — Probe #1. Runs `docker info` with a short timeout; non-zero exit or timeout → fail. No shell action (this is a stop class). Injection seam for the spawn function so tests stub Docker without touching the real CLI.
- `mcp/server/src/probes/node.ts` — Probe #2. Reads `process.version`, parses major, fails when `< 18` with the detected version in the message.
- `mcp/server/src/probes/pnpm.ts` — Probe #3. Runs `pnpm --version`; non-zero exit or ENOENT → fail. Injection seam shared with `docker.ts`.
- `mcp/server/src/probes/suiCli.ts` — Probe #4. Runs `sui --version`, parses the version string, fails outside the inclusive range `1.63.2`–`1.64.1`. Reports the detected version and the install hint (`brew install sui`). Guided stop, not hard stop. Injection seam shared with the others.
- `mcp/server/src/probes/suiPilot.ts` — Probe #5. Reads `~/.claude/settings.json` and asserts `enabledPlugins["sui-pilot@<source>"] === true` for any source. Pure filesystem read; no shell.
- `mcp/server/src/probes/sandboxRepo.ts` — Probe #6. `fsPromises.stat` against `~/workspace/deepbook-sandbox/` and asserts directory; failure message contains the exact `git clone --recurse-submodules https://github.com/MystenLabs/deepbook-sandbox.git ~/workspace/deepbook-sandbox` command from spec.md.
- `mcp/server/src/probes/manifest.ts` — Probe #7. `fetch('http://localhost:9009/manifest')` with a short connect timeout; non-200 or connect refused → fail. On fail, the probe attaches a `ShellAction` of `{ kind: 'shell', command: 'pnpm deploy-all --quick', cwd: <resolved sandbox path>, timeoutMs: 420000 }`. The action is **only attached on the fail path**; passing probes never carry shell actions. The deploy executor lives in this file as `runDeployRemediation()` which the tool calls when `remediate: true` is passed and probe-#7 has just failed; it implements the two-mode (stub vs real) branching driven exclusively by `process.env.E2E_DEPLOY_STUB === '1'`.
- `mcp/server/src/probes/learningOutputStyle.ts` — Probe #8. **Imports `probeOutputStyle` from `outputStyle.ts` and re-classifies the existing `OutputStyleResult` as a probe outcome.** No re-implementation of the settings.json read; reviewer will grep for any second copy of `learning-output-style@claude-plugins-official`.
- `mcp/server/src/tools/runPreflightProbe.ts` — New tool handler. Validates `probeId` against `PROBE_ORDER`, looks up the probe, calls `runProbe(probeId, { remediate })`. When `remediate === true` AND the underlying probe returned a `ShellAction`, it invokes the deploy executor (manifest.ts) which itself enforces the precondition gate (probes #1/#4/#6 must pass before spawn). Returns `{ pass, message, action? }` shaped as the spec.md `## MCP Tools` row describes.
- `mcp/server/src/index.ts` — Extend `registerTools(server)` to register `runPreflightProbe` alongside `start`. Tool description names the eight probe ids and notes that `remediate: true` is the only switch that triggers shell action execution. No other change to bin/exports.
- `mcp/server/src/warnings.ts` — **New** unified discriminated-union module. Exports `EngineWarning =` `RegistryWarning | StateWarning | PreflightWarning`, each tagged by literal `kind`. Existing wire `kind` values preserved verbatim (`path-malformed`, `paths-empty`, `paths-missing`, `state-corrupt`, `state-schema-mismatch`, `settings-file-missing`, `settings-parse-error`, `learning-output-style-disabled`). New preflight kinds added here: `preflight-fail` (carries `probeId`), `preflight-deploy-precondition-failed` (carries the failing probe id and is what the manifest probe emits when remediate is requested but #1/#4/#6 fail), `preflight-deploy-timeout` (real-mode only). Resolves cycle 2 carry-forwards C017+C018.
- `mcp/server/src/registry.ts` — **Edit only**: replace local `RegistryWarning` declaration with an import from `warnings.ts`. No behavior change. Wire payload identical to cycle 1.
- `mcp/server/src/tools/start.ts` — **Edit only**: replace local `StateWarning` declaration with an import from `warnings.ts`. No behavior change to the `start` tool's response shape. AC-1.3 zero-write invariant re-asserted.
- `mcp/server/src/state.ts` — **Edit**: rewrite the `saveState` durability sequence to use `fsPromises.open(tmpPath, 'r+')` → `handle.sync()` → `handle.close()` → `rename`, removing the existing `fs.openSync(..., 'r')` + `fs.fsyncSync` (cycle 2 C002+C007+C009+C027). Add `{ flag: 'wx', mode: 0o600 }` to both `fsPromises.writeFile` call sites — the tmp write in `saveState` and the archive write in `archiveCorruptFile` (cycle 2 C016). No change to `LoadStateResult`/`saveState` signatures or to the four classification kinds; cycle 2's regression tests must still pass after the rewrite.
- `scripts/e2e/harness.ts` — **Extend**: add a `runPreflightProbe(probeId, opts?)` convenience wrapper on the harness instance (just calls `callTool('runPreflightProbe', ...)`); add fixture helpers `withDockerStub({ exitCode })`, `withSuiCliStub({ version })`, `withSandboxRepoAbsent()`, `withDeployStub({ exitCode, exposeManifest })` that the harness scenarios can use to deterministically drive probes. The harness sets `E2E_DEPLOY_STUB=1` only inside the deploy-stub fixture's lifecycle; tests touching the real-deploy branch must NOT set it.
- `tests/preflight.test.ts` — Unit tests for `mcp/server/src/preflight.ts` and the per-probe modules. Each probe gets its own describe block: pass/fail output shapes, message content, fail-path shell action presence (only manifest probe has one). Asserts `PROBE_ORDER` is exactly the spec-table order (`['docker-running', 'node-version', 'pnpm-available', 'sui-cli-version', 'sui-pilot-enabled', 'sandbox-repo-present', 'sandbox-manifest-reachable', 'learning-output-style-enabled']`). Asserts probe #8 imports from `outputStyle.ts` (e.g. by mocking `probeOutputStyle` and verifying the spy was called from the probe).
- `tests/runPreflightProbe.test.ts` — Unit tests for the tool handler: rejects unknown probe ids, asserts `remediate: true` without an attached `action` is a no-op, asserts `remediate: true` with an attached action AND deploy executor execution gates on probes #1/#4/#6, asserts `E2E_DEPLOY_STUB=1` short-circuits to the stub branch and the stub branch never spawns a real subprocess (spy on `child_process.spawn` and assert zero invocations), asserts unset `E2E_DEPLOY_STUB` routes to the real branch (spy on the spawn function and assert it was called with `pnpm deploy-all --quick` and the resolved sandbox cwd — but the test mocks the spawn return so no real process actually runs).
- `tests/harness.preflight.test.ts` — Integration-shaped tests through the in-memory MCP transport for E-003, E-007, E-009, E-010, E-011 (the cycle-3 e2e scenarios that don't require real Docker). E-008 is exercised in Phase F, not in this unit suite. The harness uses the fixture helpers to stub Docker / Sui CLI / sandbox-repo / deploy without touching the real environment.
- `tests/state.test.ts` — **Extend** (not rewrite): add a regression test for the cycle-2 C002 fix — assert the fsynced FileHandle was returned by an `open` call observed before the `sync` call (relationship assertion, not call-order-only). Add a regression for the `wx` flag — assert `fsPromises.writeFile` was called with `{ flag: 'wx', mode: 0o600 }` at both the tmp and archive sites.

## Acceptance

- A1. `mcp/server/src/preflight.ts` exports `PROBE_ORDER` as an immutable, literal-typed array whose contents are exactly, in this order: `docker-running`, `node-version`, `pnpm-available`, `sui-cli-version`, `sui-pilot-enabled`, `sandbox-repo-present`, `sandbox-manifest-reachable`, `learning-output-style-enabled`. Reviewer grep target: any test or source change that re-orders this array fails the cycle. Maps spec.md `## Preflight Checks (ordered)` rows 1–8.
- A2. Probe `docker-running` passes when the injected docker-spawn returns exit code 0 and fails otherwise; failure message contains `Docker Desktop is not running` (verbatim from spec.md row 1) and the failure attaches no `action`. Failure class is "stop" — the surfaced message instructs the student to open Docker Desktop and re-run `/start`. (AC-2.4)
- A3. Probe `node-version` passes when `process.version` parses to major `>= 18` and fails otherwise; failure message names the detected version literally. (AC-2.1)
- A4. Probe `pnpm-available` passes when the injected pnpm-spawn returns exit code 0 with a parseable version and fails on ENOENT or non-zero exit; failure message contains the install hint `npm install -g pnpm`. (AC-2.1)
- A5. Probe `sui-cli-version` passes only when the parsed `sui --version` is in the inclusive range `1.63.2`–`1.64.1`; an out-of-range version (e.g. `1.62.0`) fails as a guided stop with a message containing both the detected version and `brew install sui`. The probe is a guided stop, not a hard stop — `runPreflightProbe` returns `{ pass: false, ... }` without throwing. (AC-2.5)
- A6. Probe `sui-pilot-enabled` reads `~/.claude/settings.json`, parses JSON, and asserts `enabledPlugins["sui-pilot@<source>"] === true` for at least one matching key (any source). On fail, message contains `claude plugins enable sui-pilot` verbatim. (AC-2.1)
- A7. Probe `sandbox-repo-present` passes when `~/workspace/deepbook-sandbox/` is a directory and fails otherwise (ENOENT, ENOTDIR, or stat error). On fail, the message contains the exact clone command from spec.md row 6: `git clone --recurse-submodules https://github.com/MystenLabs/deepbook-sandbox.git ~/workspace/deepbook-sandbox`. (AC-2.2)
- A8. Probe `sandbox-manifest-reachable` passes when `fetch('http://localhost:9009/manifest')` returns HTTP 200 and fails on any non-200 / connect refused / timeout. Only on the fail path does it attach `action: { kind: 'shell', command: 'pnpm deploy-all --quick', cwd: <resolved sandbox path>, timeoutMs: 420000 }`. The pass path never carries `action`. (AC-2.3)
- A9. Probe `learning-output-style-enabled` invokes `probeOutputStyle` from `mcp/server/src/outputStyle.ts` and re-classifies its result. Reviewer grep target: a search of `mcp/server/src/probes/learningOutputStyle.ts` for the literal `learning-output-style@claude-plugins-official` returns ZERO matches (the literal lives only in `outputStyle.ts`). The file `mcp/server/src/outputStyle.ts` is unchanged in this cycle.
- A10. `runPreflightProbe(probeId, ...)` rejects unknown `probeId` values with a structured error that names the probeId, never throws an uncaught exception. Known ids return whatever the underlying probe returned, modulo the remediate gate below.
- A11. `runPreflightProbe({ probeId, remediate: true })` is a no-op (returns the underlying probe's result unchanged) when the probe did not attach a shell action. Only when (a) the probe returned `pass: false` AND (b) attached an `action.kind === 'shell'` does the tool consider executing it. The tool surface keeps the contract that `action` lives on the response, not as a side effect — the caller (skill) is responsible for surfacing the command to the student before invoking with `remediate: true`.
- A12. `runPreflightProbe` is the **only** tool registered by `registerTools(server)` whose response can carry `action.kind === 'shell'`. Reviewer grep target: across the entire `mcp/server/src/tools/` directory, the literal `kind: 'shell'` (single or double quoted) appears in zero files except `runPreflightProbe.ts`. Cycle 1's A14 / A16 invariant is preserved on the new surface — `start` continues to emit no shell actions. (agent-config.md invariant 7)
- A13. The deploy executor (in `manifest.ts`) gates the real-mode spawn on probes #1, #4, and #6 returning `pass: true`. If any of those preflight preconditions fails at the moment of remediation, the executor refuses to spawn `pnpm deploy-all --quick` and returns a `preflight-deploy-precondition-failed` warning naming the failing probe. (AC-2.4 holds even mid-remediation.)
- A14. The deploy executor uses `process.env.E2E_DEPLOY_STUB === '1'` as the **single, exclusive** entry to the stub branch. No other env var, no flag on the tool params, no JSON config switch routes to the stub. Reviewer grep target: any string equal to `E2E_DEPLOY_STUB` outside the deploy executor and its tests fails the cycle. Inverse target: any other env var name used to enter the stub branch fails the cycle.
- A15. With `E2E_DEPLOY_STUB=1` set, the deploy executor takes the stub branch: it does NOT call `child_process.spawn` (test asserts spy invocations === 0), it calls the harness-injected stub instead, and `runPreflightProbe` returns the stub's result. With `E2E_DEPLOY_STUB` unset (or any other value), the stub branch is unreachable; the executor calls the real spawn function with `command='pnpm'`, `args=['deploy-all', '--quick']`, `cwd=<resolved sandbox path>`, and a `timeoutMs` of 420000.
- A16. After spawn (success exit OR timeout), the manifest probe re-runs up to three times before classifying as a final stop.
