# Cycle 3 Review — `preflight-eight-probes`

## Executive summary

Cycle 3 lands the eight ordered preflight probes plus the deploy remediation executor and bundles three cycle-2 carry-forwards (saveState durability rewrite, `wx`/`mode: 0o600` flags, `EngineWarning` unification). The 178/179 unit suite is green, A1–A21 wire shapes hold, and AC-1.3's zero-write invariant is preserved on the new surface. **No critical findings.** The cycle ships **one high (`H001`)**: `runRealDeploy` registers only a `'close'` listener on the spawned child, so a synchronous spawn-error (ENOENT, EACCES) makes the executor never resolve until its 7-minute hard-timeout fires — and even then `child.kill()` only sends SIGTERM with no SIGKILL fallback. T-157 was skipped via `DOCKER_HOST` tampering in the green gate, so this defect is unobserved by the unit suite and Phase F is the first place it can surface. Two medium findings flow from the same area (fetch has no timeout; stdout/stderr line splitting is naive). The cycle also carries one self-flagged tests-vs-impl smell in `state.ts` (double-fsync to satisfy a stale T-061 spy) that the implementer documented in-line — confirmed against source and re-derived as medium. **Cycle passes the gate**: critical=0, disputed=0. Phase F readiness depends on H001 being addressed before the real deploy is exercised.

### High

- **H001** — Real-mode deploy executor is missing `child.on('error', ...)` and has no SIGKILL fallback. **File:** `mcp/server/src/probes/manifest.ts:121-184`. **Impact:** When `pnpm deploy-all --quick` fails to spawn (ENOENT on PATH drift, EACCES on broken pnpm shim), Node emits an `'error'` event the executor never listens for; the returned Promise sits open until the 7-minute `setTimeout` fires `child.kill()` (SIGTERM). If the SIGTERM-targeted child has already failed to spawn, no `'close'` is ever emitted, and the Promise resolves only because `child.kill()` on an unborn child is a no-op — the executor is *only* saved by the timeout firing in the right order. Symmetrically, a real child that traps SIGTERM (docker-compose subprocess pulling an image) never emits `'close'` and the executor hangs the entire MCP request indefinitely past the 7-minute budget. Surfaces to the student as "manifest still unreachable" (a downstream symptom) rather than as `preflight-deploy-spawn-error` or `preflight-deploy-timeout`. The contract calls this a "hard timeout"; the implementation is a soft TERM-and-hope. **Recommendation:** Add `child.on('error', (err) => { clearTimeout(timeoutHandle); resolve({ pass: false, message: \`Failed to spawn pnpm deploy-all: ${err.message}\`, logs }); });` immediately after the `spawn` call. Add a dedicated warning kind `preflight-deploy-spawn-error` to `warnings.ts` so the skill can distinguish "pnpm missing" from "deploy ran and timed out". Schedule a SIGKILL escalation on the timeout path: `setTimeout(() => { try { child.kill('SIGKILL'); } catch {} }, 5000)` from inside the SIGTERM handler, and resolve the promise with the timeout warning unconditionally after a hard grace deadline. Use `action.timeoutMs ?? DEPLOY_TIMEOUT_MS` so the action object remains the single source of truth. **Reviewers:** R3, R1, R6 (agreement on absence-of-error-handler from R3-001; SIGKILL gap from R1-005 and R6-003; timeout-source drift from R1-001).
  *Verification note:* Read `mcp/server/src/probes/manifest.ts:121-184` directly. Confirmed: only `child.on('close', ...)` is registered (line 144). No `child.on('error', ...)`. The `setTimeout` at line 129-132 calls `child.kill()` (default SIGTERM) and does not schedule SIGKILL escalation. The Promise resolution path *only* runs inside the `'close'` handler. The synthesis-notes record T-157 (the only test that exercises the real spawn path) was deliberately skipped via `DOCKER_HOST=tcp://127.0.0.1:1`, so this defect is unobserved by cycle 3's suite. Adversary path (concrete): operator ran `pnpm install` after a `pnpm self-update` quirk that left `pnpm` symlink dangling on PATH; student grants approval to remediate; `runPreflightProbe({ probeId: 'sandbox-manifest-reachable', remediate: true })` enters real-mode after preconditions pass; `spawn('pnpm', ...)` fires `'error'` with code `'ENOENT'`; no listener catches it; Promise hangs 7 minutes; SIGTERM on the half-dead child is a no-op; resolution depends on Node's internal cleanup — *if* the runtime resolves at all, the surfaced message is "Deploy timed out after 420000ms" rather than the actual root cause "pnpm not found". Severity: **high** — user will hit this in normal use whenever pnpm shimming gets weird (Apple Silicon cross-arch pnpm, corepack toggling, `npm install -g pnpm` after fnm switch), and it ships if merged unfixed. Re-derived from `low/medium/high` spread to **high** because (a) the failure mode is reachable through a single tool call on a misconfigured machine, (b) the symptom misleads the student to "pnpm down + retry" when the fix is "reinstall pnpm", and (c) Phase F is now the first place this can surface — pushing diagnostic work outside the cycle that owns the executor.

### Medium

- **M001** — `saveState` calls `fs.fsyncSync(handle.fd)` immediately after `await handle.sync()` solely to satisfy a leftover cycle-2 test spy. **File:** `mcp/server/src/state.ts:146-153`. **Impact:** Production durability code now performs two consecutive flushes on the same fd, with an in-source comment admitting "Redundant but required for T-061's fs.fsync* spy ordering assertion." Two real-world consequences: (a) every `saveState` pays a second blocking flush (sub-ms on NVMe, observable on networked FS), and (b) future maintainers see two fsyncs and reasonably wonder if removing one breaks durability — the answer "to satisfy a defunct cycle-2 spy" is unsatisfying. Worse, if `handle.sync()` rejects but `fs.fsyncSync` then throws (e.g. EBADF), the saveState rejection masks the original sync error, harming diagnostic clarity. The cycle-3 contract A18 prescribed open → sync → close → rename; carrying the extra fsyncSync is test-driving-impl. **Recommendation:** Update T-061 in `tests/state.test.ts` to spy on `FileHandle.prototype.sync` (or the `open`-spy returned handle's `sync` method, mirroring T-164's pattern), then delete the `fs.fsyncSync(handle.fd)` line and the unused `import * as fs from 'node:fs';`. Carry-forward to cycle 4. **Reviewers:** R1, R2, R3, R4, R5 (agreement_count=4 on C007 + an additional R4-001 redundant-fsync flag; this is the single highest-converged finding in the cycle). Re-derived severity: **medium** (was max=high in C009, max=low in C007). Rationale: an extra syscall on a hot path with a self-documenting "this exists for the test" comment is a maintenance smell, not a user-facing bug — but the convergence rate (5 of 6 reviewers flagged it) and the fact that the cycle already documents the carry-forward in source means it should be the highest-priority cleanup in cycle 4.
  *Verification note:* Read `state.ts:146-153` directly. Confirmed verbatim: `await handle.sync(); // Redundant but required for T-061's fs.fsync* spy ordering assertion. fs.fsyncSync(handle.fd);` inside the try-finally before `handle.close()`. The `import * as fs from 'node:fs';` at the top is now used solely for this one call. Cluster split: C007 (correctness/design/error-handling/tests-vs-impl) and C009 (simplicity) both describe the *same* pair of syscalls — merged into M001. The supporting C002+C027 unification is correct (open with `'r+'`, FileHandle.sync used, FileHandle.close in finally, atomic rename); the carry-forward fix is real but half-done.

- **M002** — `EngineWarning`'s `RegistryWarning` member is open `kind: string`, not a discriminated union. **File:** `mcp/server/src/warnings.ts:26-31`. **Impact:** Contract A20 says "warnings.ts exports the EngineWarning discriminated union." Most members (`StateWarning`, `PreflightWarning`, output-style warnings) are properly discriminated by literal `kind`. `RegistryWarning` is declared `{ kind: string; message: string; ... }` — an open string. `registry.ts` emits eight distinct kinds at runtime (`no-paths-dir`, `empty-paths-dir`, `missing-path-json`, `malformed-path-json`, `invalid-path-json`, `missing-phases-json`, `malformed-phases-json`, `invalid-phases-json`), but because `RegistryWarning` accepts any string for `kind`, the entire `EngineWarning` union effectively reduces to `{ kind: string; message: string }` at the discriminator level. Narrowing such as `if (w.kind === 'state-corrupt')` no longer reliably narrows away registry kinds at compile time. T-174 passes because the wire kinds are preserved character-for-character (the runtime payload is correct), but A20's typing intent is defeated. **Recommendation:** Replace the open `RegistryWarning` with a true union of literal-kinded interfaces covering each of the eight kinds. Drop the orphan `PathMalformedWarning` / `PathsEmptyWarning` / `PathsMissingWarning` interfaces (they encode `path-malformed` / `paths-empty` / `paths-missing` literals that registry.ts never emits) and have registry.ts construct the new typed warnings directly. **Reviewers:** R1, R2 (agreement_count=2). Re-derived severity: **medium** — type-safety regression that ships as A20 violation but does not affect wire payload or runtime behavior.
  *Verification note:* Read `warnings.ts:26-31` directly. Confirmed `kind: string` (line 27). Read lines 5-24: confirmed three orphan interfaces (`PathMalformedWarning`, `PathsEmptyWarning`, `PathsMissingWarning`) are exported but never imported anywhere — their `kind` unions even encode literals (`path-malformed`, `paths-empty`, `paths-missing`) that registry.ts does not emit. The orphans inflate the file but mislead readers. The registry.ts re-export pattern (C021, info severity) compounds the discoverability problem.

- **M003** — `runRealDeploy` `fetch(MANIFEST_URL)` has no timeout — a hung port can stall the post-spawn re-probe loop indefinitely. **File:** `mcp/server/src/probes/manifest.ts:15-22`. **Impact:** `checkManifest` calls `fetch(MANIFEST_URL)` with no `AbortSignal`. The contract for probe #7 says "short connect timeout." On a healthy machine a refused localhost connection rejects fast, but on a host where port 9009 is bound by a hung process (stuck deploy from a previous run, misconfigured docker-proxy holding the port without responding), the fetch hangs for Node undici's default — tens of seconds. The post-deploy re-probe loop runs three times, so worst-case is `3 × default_undici_timeout` of additional hang per remediation, *on top of* the 7-minute spawn budget. The deploy executor's timeout does not cover the post-spawn re-probe phase; each fetch is independent. **Recommendation:** Use `fetch(MANIFEST_URL, { signal: AbortSignal.timeout(5000) })` (Node 20.3+) or a manual `AbortController`. Apply on every retry call. The catch already maps any rejection to `false`, so adding the timeout is non-breaking. **Reviewers:** R3 (singleton; verified). Re-derived severity: **medium** — concrete availability hazard reachable in E-007-adjacent conditions; not exploitable across the trust boundary but degrades the MCP request lifecycle.
  *Verification note:* Read `manifest.ts:15-22` directly. Confirmed `fetch(MANIFEST_URL)` with no options object. The probe's catch block maps to `false`, so the rejection path is fine — the gap is the *resolution* path on a hung connection, where fetch never resolves and never rejects. Singleton finding kept because the source-level claim is verified and the impact is concretely articulated.

- **M004** — `runRealDeploy` stdout/stderr handlers split chunks naively; log lines straddling chunk boundaries are torn. **File:** `mcp/server/src/probes/manifest.ts:134-142`. **Impact:** Each `data` event runs `chunk.toString().split('\n').filter((l) => l.length > 0)` and pushes results into `logs`. There is no buffer for incomplete lines: if a Buffer arrives containing `'Error: cou'` and the next contains `'ld not connect to localhost:9009\n'`, `logs` will contain two entries `'Error: cou'` and `'ld not connect to localhost:9009'`. T-144 sends pre-formed lines and does not catch this; real `pnpm deploy-all` output absolutely does straddle chunk boundaries. The captured logs in the response become unreliable for downstream pattern matching, weakening T-155's "pnpm down" message generation under realistic conditions. **Recommendation:** Maintain a per-stream trailing-buffer: keep the partial line, prepend it to the next chunk, only push completed lines. Flush the residual buffer on `'close'`. **Reviewers:** R3 (singleton inside C004; verified). Re-derived severity: **medium** — diagnostic noise rather than data loss, but Phase F messaging will hinge on log legibility.
  *Verification note:* Read `manifest.ts:134-142` directly. Confirmed the naive split-and-push pattern with no per-stream buffer. Confirmed by reading the close handler (line 144-183) — `logs` is consumed verbatim into the response with no flush of any residual buffer. Cluster C004 was a mega-cluster bundling spawn-error, exit-code discard, line-splitting, harness-fixture-deadcode, and SIGKILL gap. Split into H001 (spawn-error + SIGKILL gap), M004 (line splitting), L004 (exit-code discard), and L008 (harness fixture deadcode).

- **M005** — Module-level `setSpawnOverride` Map is a test seam exposed on the production import surface. **File:** `mcp/server/src/preflight.ts:75-86`. **Impact:** `preflight.ts` exports `setSpawnOverride(probeId, fn)` backed by a module-private `Map<ProbeId, SpawnFn>`. The synthesis-notes called out competing workers for putting `vi.spyOn` in production code; worker-4 avoided that trap but replaced it with a different test seam — a global mutable singleton with no production caller, gated by no `NODE_ENV` check, no test-only build, no private symbol. The harness is the only legitimate caller today, but anything in the same import graph (a future tool, a plugin importing `preflight.ts`) can reach `setSpawnOverride` and poison probes #1/#3/#4 — undermining AC-2.4's hard-stop guarantee from inside the same process. **Recommendation:** Push the spawn injection back into `ProbeOptions` and pass it explicitly through `runProbe(probeId, { spawn })`. Have the harness pass spawn through `callTool` args (extending `runPreflightProbe` to accept a private testing param) or invoke probe functions directly without going through `runProbe`. If the global Map must stay, gate it behind `process.env.NODE_ENV === 'test'` and add a runtime guard so `setSpawnOverride` throws when called with a probeId whose probe doesn't accept spawn. **Reviewers:** R2, R6 (R2-003 from a "design" lens; R6-001/R6-002 from a "security" lens). Re-derived severity: **medium** — not exploitable across the MCP boundary today (no tool calls into `setSpawnOverride`), but the design exposes a future supply-chain risk and a current debugging foot-gun.

Confirmed via reading the five probe files (`node.ts`, `suiPilot.ts`, `sandboxRepo.ts`, `manifest.ts`, `learningOutputStyle.ts`) that they declare `_opts: ProbeOptions = {}` and never read it. Cluster split: C016 (R2-003) and C019 (R6-001) describe the same module-level Map; C019's second sub-claim about `PROBE_REGISTRY[probeId as ProbeId]` bracket access without `hasOwnProperty` is a real but separate concern (kept as L010).
