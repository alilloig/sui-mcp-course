### High

- **C001** — EACCES/ENOTDIR read-error path emits `archivedTo: ''`, violating `LoadStateResult` discriminated-union contract. **File:** `mcp/server/src/state.ts:36-40` (producer); `mcp/server/src/tools/start.ts:58-65` (consumer). **Impact:** A student whose `state.json` is unreadable (permission flip, accidental `chmod 000`, ENOTDIR from a parent symlink replacement) sees a `state-corrupt` warning whose `archivedTo` field is the empty string, implying an archive exists at the empty path. Cycle 4's recovery flow cannot distinguish "bytes were unsalvageable but archived" from "I never even read the file" without grepping the message. Type-level guarantee `archivedTo: string` is already a lie. **Recommendation:** Either widen `LoadStateResult` so the corrupt arm has `archivedTo?: string` and omit it on the read-error branch, OR introduce a fourth kind `read-error` distinct from `corrupt`. Update `tools/start.ts` to omit `archivedTo` from the warning when no archive was written. Tighten T-058 to assert `archivedTo` is `undefined` (or absent) on the EACCES branch — currently it asserts only the message regex, which already passes today (see C026). **Reviewers:** R1, R3 (agreement=2; cluster also implicated in C006/R2-002 and C015/R6-005 — same defect, three review lanes saw it). *Verification note:* Read state.ts lines 27-41 directly; the assignment `archivedTo: ''` is unconditional on the non-ENOENT branch. Read start.ts lines 58-64; `stateResult.archivedTo` is forwarded into the warning without an empty-check. T-058 only matches `/read|permission|access/i` against the message, and the literal prefix "Failed to read state file:" satisfies the regex on every non-ENOENT error code regardless of any branching the impl performs (this is the C026 tautology). The test does not protect the type-level invariant.

- **C008a** — `archiveCorruptFile` failure rejects all of `loadState`, masking the primary corruption diagnostic. **File:** `mcp/server/src/state.ts:47-55, 58-69, 82-91`. **Impact:** Three branches of `loadState` call `await archiveCorruptFile(...)` with no surrounding try/catch. If the archive write itself throws — disk full (ENOSPC), permission flip on `.sui-deepbook-course/` (EACCES), or `.sui-deepbook-course` becomes a file (ENOTDIR), all of which are exactly the secondary failure modes the corruption-rescue contract was supposed to be robust against — `loadState` rejects, the SDK turns the rejection into a generic MCP tool error, and the user sees no `state-corrupt` warning at all. The original corruption fact is lost; cycle 4's recovery flow has no `kind` to dispatch on. **Recommendation:** Wrap each `await archiveCorruptFile(...)` in `try/catch` and degrade gracefully — return `{ kind: 'corrupt', archivedTo: undefined, message: 'corrupt; archive failed: ${err.message}' }` so the corruption classification still surfaces even when archival fails. Pair with C001's recommendation (make `archivedTo` optional on the corrupt arm) so the type cleanly accommodates the degraded path. Add a unit test that mocks `fsPromises.writeFile` to throw and asserts `loadState` still resolves with `kind === 'corrupt'`. **Reviewers:** R3 (singleton, but verified end-to-end). *Verification note:* state.ts:49 `const archivedTo = await archiveCorruptFile(...)` — this is the JSON-parse-failure branch; same pattern at 64 (missing schema_version) and 85 (validateState failure). Adversary path: a project where `.sui-deepbook-course/` survives the first archive but then has its perms tightened (or the device fills up while saveState is dormant) — an immutable scenario in the wild — bricks the start tool entirely instead of degrading. This is the same silent-failure class cycle 1 carry-forward C002/C010 was meant to prevent on the new surface, hence `high`.

### Medium

- **C002+C007+C009+C027** (merged) — `saveState` fsyncs an O_RDONLY fd that did not own the write; durability of the tmp file is platform-dependent. **File:** `mcp/server/src/state.ts:131-143`. **Impact:** `fsPromises.writeFile(tmpPath, ...)` opens its own internal write fd, writes, and closes it; the subsequent `fs.openSync(tmpPath, 'r')` is a *separate* read-only fd, and `fs.fsyncSync` is run on that. POSIX permits fsync on O_RDONLY but the durability guarantee for data not yet flushed from the writing fd's page cache is implementation-defined; on Windows and some network filesystems it's a no-op. After power loss the canonical `state.json` may exist as zero-length or torn — defeating the contract A7 atomicity sell. T-061 only asserts call order, not that fsync targets the writing fd, so the gap is invisible to vitest. **Recommendation:** Use `fsPromises.open(tmpPath, 'r+')` to obtain a `FileHandle`, write through it (`handle.writeFile(bytes)`), `await handle.sync()`, then `await handle.close()` — single fd, write-flagged, durably synced, uniformly async (fixes the sync-in-async smell C007 raised). Update T-061 to assert the fsynced fd was returned by an `open` call observed before the fsync call. **Reviewers:** R1 (correctness), R2 (design / async-sync mix), R3 (error-handling), R5 (tests-vs-impl). Four reviewers across four dimensions converge here; this is the strongest cross-dimension cluster in the cycle and the verification escalates the singleton severities to a stable `medium`. *Verification note:* state.ts:135 reads `const fd = fs.openSync(tmpPath, 'r')`. The flag is literally `'r'`. The contract A7 wording ("fsync(fd) on the open tmp file") was meant as an example of a write-fd sync; the implementer parsed it as "any fd referencing the tmp file." Not exploitable but the durability claim is weaker than the contract's prose suggests, and a future maintainer reading state.ts will not realize.

- **C003+C010** (merged) — Schema-validation-failure on a `schema_version=1` file is archived rather than reported as a structurally-distinct kind; same for the JSON-parseable-but-not-an-object case. **File:** `mcp/server/src/state.ts:58-70, 82-91`. **Impact:** Two contractually-undefined gray-area conditions (parseable-but-no-version-key, parseable-and-version-correct-but-shape-invalid) both archive-and-classify-as-corrupt. The contract distinguishes corrupt (parse fail) from schema-mismatch (no archive) deliberately; the gray cases are folded into corrupt-with-archive, which is defensible but means a forward-compatible schema-1.x bump that adds a required field will retroactively classify every existing state.json as corrupt and produce one archive per `start` invocation. The "do not conflate" promise the cycle's intro paragraph emphasizes is not honored on this branch. **Recommendation:** Either (a) introduce a `kind: 'shape-invalid'` / `'state-shape-invalid'` with no archive (mirrors A5's no-archive policy), or (b) bump `STATE_SCHEMA_VERSION` on every shape change and lock the choice with a comment. Preferred: option (a), with the new `kind` surfaced as a third state-warning variant. **Reviewers:** R1. *Verification note:* state.ts:64 unconditionally archives the missing-schema_version case; state.ts:85 unconditionally archives the validateState-failed case. The canonical `state.json` is preserved per A4, so the per-call archive accrual is real (linear in start invocations). Today this is `medium` because cycle 4's State has not yet evolved; the moment it does, this becomes a user-facing regression on every existing student.

- **C008b** (split from C008) — `schema_version` `typeof === 'number'` accepts `NaN`/`Infinity`/floats. **File:** `mcp/server/src/state.ts:58-79`. **Impact:** A `state.json` with `"schema_version": NaN` parses (well, `JSON.parse` itself rejects bare `NaN`, but a hand-edited `1e999` does parse to `Infinity`). The `typeof` check passes, the `!== STATE_SCHEMA_VERSION` comparison evaluates true, and the schema-mismatch warning's `foundVersion` field carries the non-finite number. Downstream `JSON.stringify(NaN)` is `null`, so the wire payload claims `foundVersion: null` while the message says `Infinity`. Confusing diagnostic; cycle 4 logic comparing `foundVersion` against an integer will silently misbehave. **Recommendation:** Replace the gate with `Number.isInteger(parsed.schema_version) && parsed.schema_version >= 1`. Treat non-integer / non-finite as corrupt-with-archive (it is malformed input, not a future schema). **Reviewers:** R3. *Verification note:* state.ts:61 uses `typeof === 'number'`, confirmed.
