# Cycle 4 Contract — `phase-engine-personalization-first-spot`

## Behavior

After this cycle, the plugin's MCP server exposes the four phase-loop tools spelled out in spec.md `## MCP Tools` — `selectPath`, `setPersonalization`, `nextSpot`, `verifySpot` — plus the load-bearing content for `paths/01-orderbook-viewer/` phase 1 (the manifest→SDK config spot at `src/App.tsx` lines 39–58). The skill drives them in the order spec.md `## Lesson Flow` prescribes: after `start` returns `outputStyleOk: true` and preflight passes, Claude renders the discovered paths and calls `selectPath({ projectRoot, slug })`; the response carries `personalizationPrompts` (the enumerable options declared in `path.json` plus a single "Use defaults" affordance per AC-6.2). Claude collects values and calls `setPersonalization({ projectRoot, values })`; the engine validates against `path.json`'s `personalization_options` declaration, persists `state.personalization` via `saveState`, and returns `{ ok, errors? }`. The skill then enters the phase loop: `nextSpot({ projectRoot })` returns the current `{ phase, spot, ladder, done }` triple; `verifySpot({ projectRoot })` runs the spot's verification adapter against the student's working tree and returns `{ pass, output?, advanced? }`. On `pass: true`, the cursor advances to the next spot (or `done: true` if the path is exhausted); on `pass: false`, the engine surfaces the failure and the next `nextSpot` call returns the same cursor so the help ladder (cycle 5) can take over.

A new `mcp/server/src/phaseEngine.ts` owns cursor progression: it reads `state.cursor` and `state.selected_path`, loads the corresponding `phases.json`, resolves the active phase/spot, and returns the substituted `Spot` view (a clone of the manifest spot with `prompt` rewritten through `personalization.ts`). Personalization substitution is **scoped exclusively to spot prompts** (and the rung markdown payloads — hint/reference/auto-write — when those are loaded; cycle 5 wires the rungs but cycle 4 establishes the substitution rule): `{{ poll_interval_ms }}` and `{{ pool_subset }}` resolve against `state.personalization`, with surrounding whitespace tolerated. The engine MUST NEVER substitute into `target_file`, `target_range`, or `verification.command`/`verification.endpoint` — those fields are emitted byte-for-byte from the manifest. AC-6.3's "personalization values surface in subsequent phase prompts as substitution and never alter the spot file/line manifest" is the load-bearing invariant; `personalization.ts` exposes a single `substitutePromptOnly(prompt, values)` entry point and reviewers will grep for any other consumer touching `target_file`/`target_range`. A new `mcp/server/src/verify.ts` owns the verification adapters: `compile` (spawns the `command` from the spot's `verification` block — for phase 1, this is `pnpm build` from `path.json`'s `build_command`, run with `cwd: projectRoot`), `test` (spawns and parses exit), `simulate` (HTTP probe matching `expected_status`), `custom` (spawns and matches `expected_stdout_regex`). Phase 1's spot uses `compile` per spec.md's phases.json example; cycle 4 wires the `compile` adapter end-to-end and ships the other three as typed stubs that throw a structured "verification mode not yet supported in cycle 4" error so future cycles flip them on without touching call sites. None of `selectPath`, `setPersonalization`, `nextSpot`, `verifySpot` may emit `action.kind === 'shell'` — `runPreflightProbe` remains the sole shell-action surface (cycle 1 A14, cycle 3 A12 carry-forward).

`paths/01-orderbook-viewer/phases.json` ships phase 1 fully populated: spot id `p1-spot-1`, `target_file: src/App.tsx`, `target_range: 39-58`, prompt + doc_links + verification, and per-rung markdown pointers (`rungs/p1-spot-1/{hint,reference,auto}.md`). Phases 2 and 3 stay schema-valid placeholders — each declares one stub spot whose prompt clearly says "TBD in cycle 5+" and whose verification mode is `compile`. The placeholders preserve AC-4.1 (every phase has at least one spot) without committing to content cycle 5+ owns. `paths/01-orderbook-viewer/phases/p1-bootstrap.md` ships the phase-1 explainer text (per spec's Lesson Flow). `paths/01-orderbook-viewer/reference/App.tsx` is a **byte-for-byte snapshot** copied from `~/workspace/deepbook-sandbox-evaluation-apps/01-orderbook-viewer/src/App.tsx` — cycle 4 must verify the source exists at apply time and SHA-256-hash both files in a regression test so silent drift between the curriculum source-of-truth and the bundled reference is caught immediately. If the source is unavailable when the cycle runs, the implementer MUST surface that as a blocker rather than guessing the content.

This cycle bundles three of the cycle-3 carry-forwards from `.forge/cycles/3/review.md`'s remediation log because their files are already in scope or trivially adjacent: **(M001)** drop `state.ts:148-150`'s redundant `fs.fsyncSync(handle.fd)` and the unused `import * as fs from 'node:fs'`, migrating cycle-2's T-061 to spy on `FileHandle.prototype.sync` (the cycle-3 review explicitly schedules this for cycle 4); **(R1-003 / R2-001)** tighten `EngineWarning.RegistryWarning` from the loose `{ kind: string; message: string }` shape into a true discriminated union covering exactly the eight kinds `registry.ts` emits (`no-paths-dir`, `empty-paths-dir`, `missing-path-json`, `malformed-path-json`, `invalid-path-json`, `missing-phases-json`, `malformed-phases-json`, `invalid-phases-json`), and drop the three orphan interfaces `PathMalformedWarning` / `PathsEmptyWarning` / `PathsMissingWarning` that encode wire kinds registry.ts never produces; **(M005 / R2-003)** retire the module-level `setSpawnOverride` Map seam in `preflight.ts` in favor of per-call `ProbeOptions.spawn` injection — the harness's `withDockerStub` / `withSuiCliStub` fixtures install the stub through a harness-internal map keyed on probe id and pass it via `ProbeOptions` at `runProbe` invocation, removing the global-mutable test seam from the production import surface. Three further cycle-3 carry-forwards are **deferred to a later cycle** to keep the AC count bounded: **R4-004** (`defaultSpawn` duplication across `docker.ts`/`pnpm.ts`/`suiCli.ts`) is pure refactor and adds no new behavior — defer; **R3-005** (`checkManifest` fetch missing AbortSignal/timeout) is a probe-internal hardening unrelated to the phase engine — defer; **C008 from cycle 2** (introduce a `state-shape-invalid` kind distinct from `state-corrupt` to handle the JSON-parseable-but-shape-invalid case without archiving) is load-bearing only the moment `STATE_SCHEMA_VERSION` bumps and the State shape evolves — and the cycle-4 State *additions* in this contract (a new `LadderRung.auto_write_attempted` field and the personalization round-trip) are append-only, so existing `schema_version: 1` files remain shape-compatible. The carry-forward stays valid for the cycle that bumps the schema version. Bundling more than three would push this contract past five carry-forward ACs and dilute the phase-engine review focus; deferring those three to cycle 5's contract or to a dedicated hardening cycle is the right tradeoff. AC-1.3's zero-write-on-output-style-fail invariant continues to hold: `selectPath`, `setPersonalization`, `nextSpot`, `verifySpot` are reachable only after `start` returned `outputStyleOk: true`, and each tool short-circuits on a `loadState` failure that classifies as `corrupt` or `schema-mismatch` (cycle 2's state recovery flow takes over before any new write). No new state writes occur on the output-style-disabled path.
