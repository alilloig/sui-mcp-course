# Phase 3 — Polling Loop with Failure Tolerance

In this phase you wire up the live order-book polling loop: a `setInterval`-backed tick that fetches mid-price and level-2 ticks from the DeepBook SDK on your chosen interval, and only surfaces an error state after 3 consecutive failures.

## What you will build

A `useEffect` block (lines 116–145 of `src/App.tsx`) that:

1. Sets up a `setInterval` with the interval configured by your `poll_interval_ms` personalization value.
2. On each tick calls `withRetry(() => Promise.all([deepbookClient.midPrice(...), deepbookClient.getLevel2TicksFromMid(...)]))` — the `withRetry` helper you built in Phase 2.
3. On success: resets the consecutive-failure counter, updates `midPrice` and `orderBookTicks` state, and clears any error.
4. On failure: increments the consecutive-failure counter, and only sets an error when the counter reaches 3 — transient single failures are silently swallowed.
5. Returns a cleanup function that calls `clearInterval` so the loop stops when the component unmounts, preventing memory leaks and stale-state updates.

## Why this matters

Polling with failure tolerance is the core user-facing behavior of the order-book viewer. Without the 3-failure gate, every sandbox container hiccup would flash an error banner. Without the `clearInterval` cleanup, navigating away from the component would leave orphaned intervals running, consuming resources and mutating unmounted component state.

## Verification

The phase verification mode is `simulate` — the engine calls `http://localhost:9009/manifest` and checks for HTTP 200 to confirm the sandbox is up and the polling tick's dependencies are satisfied.
