# Phase 2 — Resilient gRPC Simulation Calls

In this phase you will add resilience to the DeepBook gRPC calls by implementing a `withRetry` helper.

## Why retries matter

DeepBook gRPC endpoints are fast but not infallible. Network hiccups, temporary service restarts, and sandbox cold-start races can cause transient failures. A bare `await deepbook.getLevel2TicksFromMid(...)` call will throw on the first failure and propagate the error to the UI. The `withRetry` wrapper catches these transient failures and re-tries with exponential back-off before giving up.

## What you will build

A generic async helper `withRetry<T>(fn, maxAttempts, baseDelayMs)` that:

1. Calls `fn()` and returns its result if it resolves.
2. On rejection, waits `baseDelayMs * 2^attempt` milliseconds and retries.
3. Re-throws the last error after `maxAttempts` consecutive failures.

## Target

Lines 103-114 of `src/App.tsx` are reserved for this helper. The function must be defined before the polling loop in phase 3 uses it.

## Verification

`pnpm build` must succeed with the new code in place. No runtime test is needed at this stage — phase 3's simulate-verification gate will exercise the full polling path end-to-end.
