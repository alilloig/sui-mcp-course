# Phase 2 — Resilient gRPC Simulation Calls

In this phase you add a `withRetry` helper that wraps the DeepBook SDK's gRPC simulation calls so transient network errors are absorbed before they surface as UI errors.

## Why retry?

The DeepBook sandbox runs a local gRPC endpoint. During development the endpoint can hiccup — brief timeouts, in-flight restarts — and a single failure should not interrupt the user experience. Wrapping each SDK call in `withRetry` absorbs these transients without the polling loop even noticing.

## What you build

- `withRetry<T>(fn: () => Promise<T>, maxAttempts: number): Promise<T>` — a generic async retry helper at lines 103–114 of `src/App.tsx`.
- The helper loops up to `maxAttempts` times, returning on the first success and re-throwing the last error after all attempts are exhausted.

## Key concepts

- **Generic return type**: `withRetry` works with any async function, not just DeepBook calls.
- **Last-error propagation**: after exhausting attempts, the last caught error is re-thrown so the caller has a meaningful error to inspect.
- **No delay between retries**: for brevity in this sandbox setting, retries are immediate. In production you would add exponential back-off.
