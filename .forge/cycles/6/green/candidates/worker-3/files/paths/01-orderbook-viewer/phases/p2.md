# Phase 2 — Resilient gRPC Simulation Calls

Welcome to Phase 2 of the Orderbook Viewer path. In this phase you will add a `withRetry` helper that makes your gRPC calls resilient to transient node failures.

## Why retries are needed

The DeepBook gRPC `SimulateTransaction` endpoint occasionally returns without `commandResults` at block boundaries or when the Sui node is temporarily busy processing a batch. Without retries, a single transient failure surfaces immediately as an error in the UI, creating a poor user experience even when the node recovers within milliseconds.

## What you are building

A generic `withRetry<T>(fn: () => Promise<T>, retries = 2): Promise<T>` helper function (lines 103–114 of `src/App.tsx`) that:

1. Attempts to call `fn()` up to `retries` times.
2. Returns immediately on the first success.
3. Waits 300ms between attempts (not before the first) to give the node time to recover.
4. Re-throws on the final failure so the caller can decide what to do.

## Acceptance

Phase 2 is complete when `pnpm build` exits 0. The TypeScript compiler will verify that `withRetry` is correctly generic-typed and that it is usable as `withRetry(() => client!.deepbook.midPrice(pool))`.
