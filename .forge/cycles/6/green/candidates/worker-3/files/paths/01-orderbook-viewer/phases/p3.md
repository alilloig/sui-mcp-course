# Phase 3 — Polling Loop with Failure Tolerance

Welcome to Phase 3 of the Orderbook Viewer path. In this phase you will implement the `tick` function and polling loop that drives live order book updates in the UI.

## What you are building

The core polling loop (lines 116–145 of `src/App.tsx`) runs inside a `useEffect` and:

1. **Fetches concurrently** — calls `withRetry(() => client!.deepbook.midPrice(pool))` and `withRetry(() => client!.deepbook.getLevel2TicksFromMid(pool, 10))` via `Promise.all` to minimize latency.
2. **Updates UI state** — on success, resets the `failures` counter to 0, updates `book` with the new mid price and ask/bid arrays, and clears any displayed error.
3. **Tolerates failures** — on catch, increments `failures` but only calls `setErr(...)` after 3 consecutive failures. Transient blips (a single missed tick) never appear as UI errors.
4. **Cancels cleanly** — checks `cancelled` before every state update; the effect cleanup sets `cancelled = true` and calls `clearInterval(id)` to stop the loop.
5. **Polls on interval** — starts with an immediate `tick()` call then repeats via `setInterval(tick, <poll_interval_ms>)` where the interval is your personalization choice.

## Why failure tolerance matters

Real DeepBook nodes see brief gaps in `SimulateTransaction` responses at epoch boundaries and during high-load moments. Surfacing every transient error to the user as a visible message would make the app feel unstable. Requiring 3 consecutive failures before showing an error gives the `withRetry` helper from Phase 2 time to recover, and only escalates genuine connectivity problems to the UI.

## Acceptance

Phase 3 is complete when the sandbox manifest endpoint (`http://localhost:9009/manifest`) is reachable and the polling loop returns valid `midPrice` and `ticks` data on the first tick.
