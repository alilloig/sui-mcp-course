# Phase 2 — Resilient gRPC Simulation Calls

Welcome to Phase 2 of the Orderbook Viewer path. In this phase you will make your DeepBook SDK calls resilient to transient gRPC failures by implementing a `withRetry` helper.

## What you are building

The DeepBook sandbox exposes its order book data via gRPC endpoints. Under realistic conditions — especially during sandbox startup, network fluctuations, or high load — individual calls can fail transiently. Without retry logic, a single failed tick would propagate as an error to the user.

Your task is to implement a generic `withRetry<T>` function that:

1. Accepts any `() => Promise<T>` callback and a `retries` count (default 3).
2. Attempts the call and returns immediately on success.
3. On failure, waits a short time and retries up to `retries` times.
4. Re-throws the error after all retries are exhausted.

## Why this matters

The DeepBook v3 gRPC client can receive `UNAVAILABLE` or `DEADLINE_EXCEEDED` status codes during startup or under load. Wrapping each call in `withRetry` means the polling loop tolerates short outages without surfacing errors to the user every few seconds.

## Acceptance

Phase 2 is complete when `pnpm build` exits 0, confirming that `withRetry` compiles with the correct generic signature and is callable from the polling loop in Phase 3.
